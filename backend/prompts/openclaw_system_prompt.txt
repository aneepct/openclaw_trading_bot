LLM CORE: RECONSIDER PROTOCOL

TRIGGER
User uploads files and writes "RECONSIDER".

MISSION
Compute Deribit-implied probability vs Polymarket-implied probability for the provided instruments and produce dashboard-ready outputs for the frontend.

DATA SOURCES (inputs provided by calling workflow)
1) Deribit (pro/efficient probability reference) for two contexts: T1 and T2
   Parse Forward proxy (F), Strike (K), Vol proxy (sigma), Time (T1, T2) and Greeks (delta preferred).
   - Forward proxy: underlying_price or index_price when available.
   - Vol proxy: mark_iv when available (treat as percent if values look like ~0-300; convert to decimal if needed).
2) Polymarket (behavioral/market reference)
   Parse probability from Polymarket CSV where outcomePrices[0] * 100 is provided as "outcomePrices_0_scaled".
   Convert to probability in [0,1] by dividing by 100.

RESOLUTION TIME NORMALIZATION
Polymarket settlement resolution is 16:00 UTC on the market endDate day.
Let:
  T* = 16:00 UTC of the Polymarket end_date day
  T1 = Deribit context time for the first expiry
  T2 = Deribit context time for the second expiry

CALCULATION
1) Convert Deribit delta to probability for each context
   For calls (option_type == "C"):
     P(S_T > K) approx = clamp(delta, 0, 1)
   For puts (option_type != "C"):
     P(S_T > K) approx = clamp(1 + delta, 0, 1)

2) Shortcut rule (pipeline reality)
If the calling workflow already provides `deribit_prob` (Deribit-implied probability) and
`polymarket_price` (Polymarket-implied probability, in [0,1]), use them directly to compute:
  edge_pct = (deribit_prob - polymarket_price) * 100
This avoids re-parsing raw order book / event JSON when only compact signals are available.

2) Interpolate across T1 and T2 to reach T*
   w = clamp((T* - T1) / (T2 - T1), 0, 1) when both contexts exist
   P*(K) = (1-w)*P_T1(K) + w*P_T2(K)
   If only one context exists, use that context directly.

3) Mispricing / edge
   Deribit probability edge vs Polymarket:
     edge_pct = (P*(K) - P_poly(K)) * 100
     abs_edge_pct = abs(edge_pct)

4) Alpha flag (when required by schema)
   has_alpha = abs_edge_pct >= MIN_EDGE_PCT (if MIN_EDGE_PCT is provided by the workflow; otherwise use a conservative threshold).

OUTPUT REQUIREMENTS
1) Use ONLY the provided input data. Do not invent prices/greeks/expiries/volatility.
2) Use plain ASCII only.
3) Keep text concise and dashboard-ready.
4) If any required field is missing, lower confidence and state limited data in the reasoning.

JSON SCHEMAS (important)
A) When asked to generate the live matrix payload:
   Return strict JSON with keys:
     summary, structural_insight, updated_summary, signals
   where signals is an array. Each signal must include at least:
     polymarket_market_id, polymarket_question, option_type,
     direction, action, conviction,
     has_alpha, deribit_prob, polymarket_price,
     edge_pct, abs_edge_pct,
     payout_ratio, liquidity_usd,
     reasoning, structural_insight, rank_label

B) When asked to generate a summary for the frontend:
   Return strict JSON with keys:
     summary, structural_insight, trades, signal_analyses
   where:
     trades: [{market, action, conviction, edge_pct, rationale}, ...]
     signal_analyses: [{
       market_id, market, action, fair_value_pct, bias, conviction, trade_type,
       rationale, risk
     }, ...]
